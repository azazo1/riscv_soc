# M 扩展资源超限记录

## 现象

在尝试把 RV32M 乘除法扩展直接放进组合 ALU 后, Quartus 布局布线出现资源超限:

```text
Error (170012): Fitter requires 21980 LABs to implement the design, but the device contains only 3207 LABs
```

这个结果说明综合后的逻辑面积已经远远超过 DE1-SoC 当前 FPGA 器件容量. 这类问题不是简单降低时钟频率就能解决, 因为首先面积已经放不下.

## 主要原因

RV32M 包含下面这些指令:

| 指令 | 作用 |
| --- | --- |
| `MUL` | 乘法低 32 bit |
| `MULH` | 有符号乘法高 32 bit |
| `MULHSU` | 有符号乘无符号高 32 bit |
| `MULHU` | 无符号乘法高 32 bit |
| `DIV` | 有符号除法 |
| `DIVU` | 无符号除法 |
| `REM` | 有符号取余 |
| `REMU` | 无符号取余 |

其中最危险的不是普通乘法, 而是组合除法和组合取余.

如果在 ALU 这种纯组合模块里直接写:

```verilog
result = lhs / rhs;
result = lhs % rhs;
result = $signed(lhs) / $signed(rhs);
result = $signed(lhs) % $signed(rhs);
```

综合器会尝试生成一个能在一个周期内完成 32 bit 除法或取余的组合逻辑网络. 这个网络通常由很多级比较, 减法和选择器组成, 面积非常大.

如果同时支持 `DIV`, `DIVU`, `REM`, `REMU`, 综合器还可能生成多套类似逻辑, 再用 mux 选择输出. 这会让资源占用快速膨胀.

## 大致资源判断

| 运算 | 资源风险 | 说明 |
| --- | --- | --- |
| 32 bit 加减法 | 低 | FPGA 对加法链支持较好 |
| 32 bit 逻辑运算 | 低 | AND, OR, XOR 都很小 |
| 32 bit 移位 | 中 | 桶形移位器需要一些 mux |
| 32 bit 乘法 | 中 | 可以映射到 DSP 时 LAB 压力较小 |
| 32 bit 组合除法 | 很高 | 一周期完成会生成大面积组合网络 |
| 32 bit 组合取余 | 很高 | 通常和除法同级别 |

因此这次超限的主要嫌疑是 `/` 和 `%`, 不是 `+`, `-`, `&`, `|` 这类基础 RV32I 运算.

## 当前处理原则

当前阶段保持硬件 CPU 为 RV32I, 不实现硬件 M 扩展.

C 代码仍然可以写乘除法表达式, 但编译目标必须限制为 RV32I. 编译器会把整数乘除法降级为软件 helper, 例如:

| helper | 作用 |
| --- | --- |
| `__mulsi3` | 32 bit 乘法 |
| `__divsi3` | 32 bit 有符号除法 |
| `__udivsi3` | 32 bit 无符号除法 |
| `__modsi3` | 32 bit 有符号取余 |
| `__umodsi3` | 32 bit 无符号取余 |

这样做的代价是乘除法运行速度变慢, 但硬件资源占用会明显下降, 更适合当前先把 SoC 跑通.

## 后续如果需要 M 扩展

后续如果确实需要硬件乘除法, 不建议重新放回组合 ALU.

更合适的实现方式是新增一个多周期乘除法单元:

- `start`: core 发起一次乘除法请求.
- `op`: 选择 `MUL`, `DIV`, `REM` 等操作.
- `lhs`, `rhs`: 两个操作数.
- `busy`: 单元正在计算.
- `done`: 结果有效.
- `result`: 计算结果.

core 遇到 M 指令时暂停 PC 和寄存器写回, 等 `done` 后再把 `result` 写入 rd.

这样面积会比组合除法小很多, 代价是每条乘除法指令需要多个周期. 对当前项目来说, 这个取舍更合理.

## 当前结论

在 DE1-SoC 当前 FPGA 资源规模下, RV32M 不能直接用纯组合 `/` 和 `%` 实现. 当前应回到 RV32I, 通过软件 helper 支持 C 语言中的乘除法. 硬件 M 扩展以后再按多周期执行单元重新设计.
